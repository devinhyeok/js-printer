## 요약

JS Printer는 Vite + React SPA에서 MDX를 사용하는 프로젝트다.
일부 콘텐츠만 비밀번호로 잠그고, 개발자 도구로도 내용을 볼 수 없게 하고 싶다는 요구가 있었다.

결론부터 말하면, 이 프로젝트 구조에서는 안 된다.
MDX, Vite 모듈 시스템, 부분 암호화 세 가지가 동시에 충돌하고, 이 조합을 해결한 레퍼런스도 없다.

## 원래 하려던 것

1. 카탈로그에서 특정 콘텐츠에 `password`를 설정
2. 빌드할 때 해당 콘텐츠를 AES로 암호화해서 번들에 평문이 안 들어가게 함
3. 브라우저에서 비밀번호 입력하면 복호화해서 보여줌

이렇게만 들으면 단순해 보이지만, MDX 프로젝트에서는 여러 곳에서 막힌다.

## 비슷하지만 다른 것들

| 방식                                   | 되는가? | 설명                                                      | 대표 도구             |
| -------------------------------------- | ------- | --------------------------------------------------------- | --------------------- |
| HTML 파일 통째로 암호화                | O       | 빌드된 HTML 전체를 암호화하고 복호화 폼 붙여서 배포       | StatiCrypt, PageCrypt |
| 일반 `.md` 파일 암호화                 | O       | 텍스트 파일이니까 암호화 → 복호화 → markdown-to-html 렌더 | 직접 구현 가능        |
| 서버에서 인증 후 콘텐츠 내려주기       | O       | 로그인 안 하면 애초에 콘텐츠를 안 보내니까 안전           | Next.js 등            |
| MDX 파일을 부분적으로 빌드 타임 암호화 | X       | 이 문서에서 설명하는 문제                                 | 없음                  |

핵심 차이: 일반 `.md`는 텍스트 파일이고, MDX는 빌드하면 JavaScript 코드가 되는 파일이다.

## 왜 안 되는가

### MDX는 텍스트가 아니라 JavaScript로 변환된다

일반 마크다운은 단순하다. 텍스트 파일이니까:

```
"안녕하세요" → 암호화 → "aGVsbG8=" → 복호화 → "안녕하세요" → HTML로 변환 → 렌더링
```

MDX는 빌드할 때 `@mdx-js/rollup` 플러그인이 React 컴포넌트 코드로 바꿔버린다:

```
# 제목     →  (빌드)  →  jsx("h1", {}, "제목")  →  번들에 JS로 들어감
```

그래서 두 가지 선택지가 생기는데, 둘 다 문제가 있다:

- 원본 MDX 텍스트를 암호화하면, 복호화 후 브라우저에서 MDX를 다시 컴파일해야 한다. MDX 컴파일러는 무거워서 브라우저에서 돌리기 비현실적이다.
- 빌드된 JS 코드를 암호화하면, 복호화 후 그 JS를 실행해야 하는데, `eval`이나 동적 `import`를 써야 한다. 보안 정책(CSP)에 걸릴 수 있고, 애초에 보안을 위해 암호화하는 건데 `eval`을 쓰는 건 모순이다.

### MDX 파일 하나가 아니라 여러 파일이 서로 엮여 있다

JS Printer의 문서 하나는 이런 구조다:

```
index.mdx
├── import Cover from './cover.mdx'
├── import Intro from './intro.mdx'
├── import Chapter01 from './chapter01/index.mdx'
│   └── import Section1 from './section1.mdx'
└── import Chapter02 from './chapter02/index.mdx'
```

`index.mdx` 하나를 암호화해봤자 `cover.mdx`, `intro.mdx` 등은 번들에 평문으로 남는다. 실제 내용은 다 거기에 있으니 의미가 없다.

그렇다고 하위 파일 전부를 암호화하면 또 다른 문제가 생긴다. 빌드된 `index.mdx`의 JS 코드 안에는 `import Cover from './cover.mdx'` 같은 구문이 들어있다. 이건 Vite가 빌드할 때 모듈 간 연결을 만들어주는 것인데, 암호화하면 이 연결이 끊어진다. 복호화한 JS 코드에 `import Cover from './cover.mdx'`가 있어도, 브라우저에서는 이미 빌드가 끝난 상태라 이 import를 다시 해석할 방법이 없다. 복호화된 `index.mdx`가 cover를 불러오라고 하는데, 브라우저는 그게 어디 있는 건지 알 수 없는 상황이다.

### 기존 코드가 React 컴포넌트를 기대하는데, 암호화하면 문자열이 나온다

현재 `DocViewer`는 MDX 파일을 불러오면 React 컴포넌트가 나올 것으로 기대하고, 바로 `<Doc />`처럼 렌더링한다.

암호화를 적용하면 React 컴포넌트 대신 `"aGVsbG8gd29ybGQ=..."` 같은 암호문 문자열이 반환된다. 문자열은 React 컴포넌트가 아니니까 `<Doc />`으로 렌더링할 수 없다.

암호화된 건 문자열로 취급하고, 복호화 후 다시 컴포넌트로 변환하는 분기 로직을 만들어야 하는데, 이게 위의 하위 모듈 의존성 문제와 겹치면서 풀 수 없게 된다.

### Vite 플러그인으로 시도해봤지만, 일부만 동작했다

실제로 Vite `transform` 훅으로 MDX의 빌드 결과를 가로채서 암호화하는 플러그인을 만들어 테스트했다.

결과:

- `index.mdx`의 암호화는 성공했다. 빌드 로그에도 찍혔다.
- 하지만 `cover.mdx`, `intro.mdx` 등 하위 파일은 transform 훅이 호출되지 않았다.
- 설령 하위 파일까지 전부 암호화에 성공하더라도, 모듈 간 import 관계 복원 불가 문제는 해결할 수 없다.

## 가능한 대안

### 사이트 전체 암호화 (StatiCrypt 방식)

빌드 완료 후 `dist/index.html` 전체를 StatiCrypt로 암호화한다.
단, 이러면 모든 콘텐츠가 잠기므로 일부만 잠금이 불가능하다.

### MDX를 버리고 일반 Markdown 사용

잠금 대상 콘텐츠만 일반 `.md`로 작성하고, 런타임에 `remark`로 렌더링하는 방식.
대신 MDX의 React 컴포넌트 기능(`<DocPage>`, `<Cover>` 등)을 사용할 수 없게 된다.

### 서버 기반 보호 (Next.js 등)

Vite SPA를 Next.js로 전환하고, 보호 대상 콘텐츠는 서버에서 인증 후 응답.
가장 정석적이고 확장 가능하지만, 프로젝트 구조를 바꿔야 한다.

### 빌드를 두 벌로 나누기

- 공개용 빌드: 잠금 콘텐츠 제외
- 전체 빌드: 비밀번호로 보호된 별도 URL에 배포

암호화가 아니라 배포 분리로 접근하는 방식이다.

## 해결책: Next.js + Supabase로 전환

위 대안들 중 가장 현실적이고 확장 가능한 방법은 Next.js + Supabase 조합으로 프로젝트를 전환하는 것이다.

### 왜 이 조합이면 되는가

| 현재 (Vite SPA) 문제                            | Next.js + Supabase 해결                            |
| ----------------------------------------------- | -------------------------------------------------- |
| 서버가 없어서 클라이언트에 모든 콘텐츠가 내려감 | API Route에서 인증 확인 후에만 콘텐츠 응답         |
| MDX를 암호화해야 하는 복잡성                    | 서버에서 MDX를 렌더링하므로 암호화 자체가 불필요   |
| 개발자 도구로 평문 노출                         | 인증 안 된 요청에는 콘텐츠를 아예 안 보냄          |
| 비밀번호 변경하면 재빌드 필요                   | DB에서 비밀번호만 바꾸면 즉시 반영                 |
| 결제 연동 불가                                  | Supabase Auth + 결제 상태 컬럼으로 자연스럽게 확장 |

### 구체적 구조

- Next.js App Router: 기존 React 컴포넌트(DocLayout, BookSwitcher, FloatingNav 등) 그대로 재사용
- MDX 렌더링: `next-mdx-remote` 또는 `@next/mdx`로 서버 사이드 MDX 처리
- Supabase: 콘텐츠별 비밀번호 저장, 회원 인증, 결제 상태 관리
- API Route: 비밀번호 확인 또는 로그인 확인 → 통과 시에만 MDX 콘텐츠 응답

### 재사용 가능한 기존 코드

- `src/components/layout/` (DocLayout, DocPage, SlideLayout, ViewerShell 등)
- `src/components/ui/` (BookSwitcher, FloatingNav)
- `src/content/` (MDX 콘텐츠 파일 전체)
- `src/styles/` (global.css, Tailwind 설정)

## 결론

동적 웹사이트에서 MDX 컴포넌트를 부분적으로 빌드 타임 암호화하는 것은:

- 기존 도구(StatiCrypt, PageCrypt)가 해결하는 문제와 다르고
- MDX 컴파일, Vite 모듈 시스템, React 컴포넌트 로딩이 동시에 충돌하며
- 참고할 레퍼런스나 선례가 없는 조합이다

이 프로젝트에서 콘텐츠 보호가 필요하다면, Next.js + Supabase로 전환하는 것이 가장 현실적이고 확장 가능한 해결책이다.
