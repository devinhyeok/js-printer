## 요약

JS Printer는 MDX 콘텐츠를 A4 문서와 16:9 슬라이드로 렌더링하여 PDF로 출력하는 프로젝트다.
shadcn/ui를 도입하면 CSS 관리가 깔끔해지지 않을까 검토했다.

결론부터 말하면, 이 프로젝트에는 안 맞는다.
CSS 복잡도의 대부분이 UI 컴포넌트가 아니라 인쇄/출력 도메인에 있고, shadcn이 해결해주는 영역이 거의 없다.

## CSS 관리 도구의 두 종류

| 종류             | 예시                               | 해결하는 문제                       |
| ---------------- | ---------------------------------- | ----------------------------------- |
| 스타일링 시스템  | Tailwind, CSS Modules, Emotion     | CSS를 어떻게 작성하고 관리할 것인가 |
| UI 컴포넌트 키트 | shadcn, MUI, Chakra UI, Ant Design | 버튼·모달·폼 같은 UI를 직접 안 만듦 |

이 프로젝트는 스타일링 시스템(Tailwind CSS v4)은 이미 적절히 사용 중이다.
검토한 것은 UI 컴포넌트 키트 도입 여부다.

## 왜 안 맞는가

### CSS 복잡도가 UI가 아니라 출력 도메인에 있다

`global.css`의 구성을 보면:

| 영역                          | 대략 줄 수 | shadcn 관련성 |
| ----------------------------- | ---------- | ------------- |
| 슬라이드 타입 시스템 (8종)    | ~220줄     | 없음          |
| 인쇄/PDF `@media print`       | ~60줄      | 없음          |
| paged.js 웹 미리보기          | ~80줄      | 없음          |
| 코드블록·수식·Mermaid         | ~40줄      | 없음          |
| 웹 화면 레이아웃 (doc, slide) | ~90줄      | 없음          |

CSS 전부가 `data-type` 기반 슬라이드 스타일, `@page` 규칙, `@media print`, paged.js 연동 같은 출력 전용 스타일이다. shadcn을 도입해도 이 영역은 1줄도 줄어들지 않는다.

### UI 표면적이 너무 작다

shadcn이 대체할 수 있는 UI 컴포넌트가 프로젝트 전체에서 이것뿐이다:

- 버튼 5~6개 (홈, 줌 인/아웃, 리셋, 맞춤, PDF 저장)
- 입력 필드 2개 (BookSwitcher 검색, ContentSelector 검색)
- 사이드바 패널 2개 (BookSwitcher, FloatingNav)

일반 웹 앱은 모달, 드롭다운, 폼, 테이블, 토스트, 탭 등 수십 종의 UI 패턴이 필요해서 키트가 효율적이다. 이 프로젝트는 그런 UI가 없다.

shadcn 도입 시 추가되는 것:

- `@radix-ui/*` 의존성 여러 개
- `lib/utils.ts` (`cn()` 유틸리티)
- `components.json` 설정 파일
- CSS 변수 기반 테마 시스템 (`--background`, `--foreground` 등)

버튼 5개와 입력 필드 2개를 위해 이 인프라를 깔면 오버엔지니어링이다.

### 인쇄 환경에서 CSS 변수가 깨질 수 있다

`react-to-print`은 별도 iframe을 생성해서 콘텐츠를 렌더링한다. `ViewerShell`의 `pageStyle`이 그 iframe 안에서 적용되는 인쇄 전용 CSS다.

shadcn 컴포넌트는 `hsl(var(--primary))` 같은 CSS 변수에 의존한다. iframe 내부에서 이 변수들이 정의되지 않으면 스타일이 전부 깨진다.

현재 코드가 `#1a1a1a`, `#6b7280` 같은 하드코딩 값을 쓰는 건 의도적이다. 인쇄 iframe 안에서도 확실하게 동작해야 하기 때문이다.

```css
/* ViewerShell의 pageStyle — iframe 내부에서 적용됨 */
body {
  font-family: 'Noto Sans KR', sans-serif;
  color: #1a1a1a;
}
```

## 현재 구조가 적절한 이유

이 프로젝트의 스타일링은 이미 역할이 깔끔하게 나뉘어 있다:

| 역할                   | 위치                        | 방식                     |
| ---------------------- | --------------------------- | ------------------------ |
| 출력 도메인 스타일     | `global.css`                | 순수 CSS (data-type 등)  |
| 인쇄 전용 스타일       | `ViewerShell.tsx` pageStyle | CSS 문자열 (iframe 주입) |
| UI 컴포넌트 스타일     | 각 `.tsx` 파일              | Tailwind 유틸리티 클래스 |
| 콘텐츠별 커스텀 스타일 | `content/*/theme.tsx`       | 콘텐츠 로컬 스타일       |

## 나중에 도입을 고려할 시점

다음 조건이 생기면 그때 검토해도 늦지 않다:

- 설정 페이지, 대시보드 등 복잡한 UI 화면이 추가될 때
- 모달, 드롭다운, 탭, 폼 등 인터랙티브 UI 패턴이 5종 이상 필요할 때
- 다크 모드 같은 테마 시스템이 필요할 때

## 결론

shadcn/ui는 UI 컴포넌트가 많은 범용 웹 앱에 적합한 도구다.
이 프로젝트는 UI가 아니라 출력이 핵심이고, CSS 복잡도가 인쇄/슬라이드 도메인에 집중되어 있다.
Tailwind CSS v4만으로 현재 요구사항을 충분히 커버하고 있으며, 추가 UI 키트는 도입 비용 대비 효과가 없다.
